# Los Santos RED Co-op v1 Manual Test Matrix

Use this matrix for practical RAGECOOP co-op validation. Mark each scenario Pass or Fail and attach the relevant client/server logs when a result is ambiguous.

| # | Test name | Setup | Expected result | Likely logs to inspect | Pass/Fail |
|---|---|---|---|---|---|
| 1 | Normal single-player launch with co-op disabled | Start GTA/RPH without the RAGECOOP co-op bridge state file or with co-op disabled. Press Shift+F10. | Los Santos RED starts exactly like normal single-player. No co-op permission restrictions, no RAGECOOP dependency, and no co-op menu changes are visible. | RPH log for normal LSR startup; absence of `[LsrCoop.Client]`; absence of `LSR co-op is waiting for active host`. |  |
| 2 | RAGECOOP server starts and loads world save | Start the RAGECOOP server with the LsrCoop server resource enabled and an existing world save present. | Server loads the shared world save, roles, profiles, and persistent records without creating duplicate world state. | `[LsrCoop.Server] role config loaded`; world/profile store load messages; safe file store backup fallback warnings if applicable. |  |
| 3 | Admin joins | Connect a player whose profile id is listed as Admin but not necessarily TrustedHost. | Player registers as Admin. Admin permissions are available only for co-op admin-scoped actions; Admin alone does not imply active simulation host unless also TrustedHost. | `[LsrCoop.Server] client registered`; role text including `Admin`; client compatibility status. |  |
| 4 | TrustedHost joins and becomes active host | Connect a compatible player whose profile id is listed as TrustedHost. | Server selects that connected compatible TrustedHost as active host. Shift+F10 on that client allows full LSR simulation startup. | `[LsrCoop.Server] active host assigned`; `[LsrCoop.Client] active host assigned`; LSR bridge state file values for `IsActiveHost=true`. |  |
| 5 | Normal Player joins | Connect a compatible player who is not Admin and not TrustedHost. | Player receives profile/session state but does not become active host. Shift+F10 does not start full LSR simulation on that client. | `[LsrCoop.Server] client registered`; role text `Player`; client log for waiting/no active host or non-host state. |  |
| 6 | New player creates character using LSR customizer flow | Connect a player with no existing profile/character. | Server marks character creation required. Client uses the existing LSR customization flow where available. Final saved character is stored to the server profile. | `lsrcoop.character.createRequired`; `[LsrCoop.Client]` character-create message; appearance/action result logs; world save profile entry. |  |
| 7 | Returning player loads one character profile | Connect a player with an existing saved profile and appearance. | Exactly one character profile loads for that player. Appearance/profile snapshot is delivered without opening forced creation. | `[LsrCoop.Server] client registered`; profile snapshot/world snapshot logs; absence of `character.createRequired` for that profile. |  |
| 8 | Player changes clothes using existing LSR behavior | In co-op, use the existing LSR clothing/customization interaction for a normal player. Apply a valid clothing change. | Preview remains local before Apply. Accepted Apply saves and syncs appearance through co-op profile state. Single-player clothing behavior remains unchanged outside co-op. | Appearance sync logs; gameplay action result for `SaveAppearance`; world save profile appearance fields. |  |
| 9 | Purchase item from LSR store | Active host is running simulation. A player buys an item from an existing LSR store using existing UI/rules. | Existing LSR requirements decide the purchase. Accepted purchase updates money/inventory and persists through the server. Rejected purchase resyncs/reverts pending state. | Gameplay action request/result logs for `PurchaseItem`; server commit logs; inventory/money snapshot fields. |  |
| 10 | Money/inventory persists | Complete a valid money/inventory change, disconnect all clients, restart server, and reconnect the same profile. | Saved money and inventory reload from the server-world profile. No direct trading or extra inventory behavior appears. | World save file; world/profile store load logs; gameplay action result logs; client profile snapshot. |  |
| 11 | Player commits crime | Active host is running simulation. One co-op player commits a normal LSR crime against NPC/world state. | Crime event carries the offender profile/character context and feeds existing LSR violation/reporting behavior. | Crime routing logs; violation/criminal justice snapshot commits; active host client logs. |  |
| 12 | PvP triggers normal LSR crime behavior | One co-op player attacks another co-op player while co-op is enabled. | PvP is routed as normal violent LSR behavior with offender/victim context. No separate PvP rule system is used. | `[LsrCoop.Client] PvP crime assigned to active host`; `[LsrCoop.Server] PvP crime routed to active host`; crime actor/victim payload details. |  |
| 13 | Gang reputation changes apply to actor | Trigger an existing LSR gang reputation or relationship change for one player. | Reputation change applies to the actor player only and is saved under that server profile. Other players' reputation records are not overwritten. | Gang reputation snapshot/commit logs; profile save data for the actor; compare another player profile entry. |  |
| 14 | Player buys/designates owned vehicle | Use existing LSR owned vehicle purchase/designation/save behavior for one player. | Vehicle ownership applies to that actor profile and persists in server-world save. Existing LSR unlock/blip behavior is preserved. | Gameplay action result for `PurchaseVehicle` or `SaveOwnedVehicle`; owned vehicle profile fields; client ownership/blip sync logs. |  |
| 15 | Player buys property if supported | Use an existing LSR property purchase path where the current XML/config supports it. | Active host validates with existing LSR property requirements. Property ownership persists for the actor profile. No crew/shared ownership is added. | Gameplay action result for `PurchaseProperty`; property profile fields; existing LSR property logs if present. |  |
| 16 | One player dies/arrests, others continue | With at least two connected players, have one player enter existing LSR wasted/busted/arrest flow. | Only the affected player enters the death/arrest flow. Other players remain in session. | 17 | Active host leaves, soft handoff occurs | With two compatible TrustedHosts connected, disconnect the active host. | Server saves persistent state, discards temporary live incidents, selects another compatible TrustedHost, and broadcasts a world snapshot. | `[LsrCoop.Server] active host released`; `active host soft handoff complete`; `lsrcoop.world.snapshot`; `[LsrCoop.Client] active host assigned`. |  |
| 18 | Server restarts, persistent state loads | After persistent changes, stop and restart the RAGECOOP server, then reconnect players. | World id, roles, profiles, appearance, money, inventory, weapons, vehicles, properties, gang reputation, and criminal history load. | 19 | Mismatched client cannot become active host | Connect a TrustedHost with mismatched co-op build/config/LSR version or missing required resource report. | Server marks client incompatible and does not assign them active host. | 20 | No temporary chase/roadblock persists after all players leave | Start a chase or temporary police incident, then disconnect all players and restart the server.
